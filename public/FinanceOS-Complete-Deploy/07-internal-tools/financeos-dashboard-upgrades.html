1. SOX Audit Trails View

Not Built Business Tier

UI Components to Build

  • AuditTrailView — new view in page.jsx (~300 lines)
  • Audit event log table with filters (date, user, action type)
  • Change diff viewer — shows before/after for data modifications
  • Export button — CSV/PDF audit report generation
  • Compliance status header with SOX readiness score
  • User action timeline visualization

Backend Requirements

  • Supabase table: audit_events (id, user_id, action, entity_type, entity_id, old_value, new_value, ip, timestamp)
  • Middleware: audit logger wrapping all data mutation endpoints
  • API route: GET /api/audit — paginated, filterable event log
  • Row-level security: users can only see their org's audit events

Data Model

  • audit_events table with JSONB columns for old_value/new_value
  • Immutable append-only — no UPDATE/DELETE allowed
  • Indexes on timestamp, user_id, entity_type for fast queries
  • Retention policy: 7 years for SOX compliance

Effort Estimate

  • UI View: 2-3 days
  • Supabase table + RLS: 0.5 day
  • Audit middleware: 1-2 days
  • API route + pagination: 1 day
  • Total: ~5-7 days

2. SSO / SAML Configuration View

Not Built Business Tier

UI Components to Build

  • SSOView — either a new view or a panel within Settings/Admin
  • Identity Provider setup wizard (Okta, Azure AD, Google Workspace)
  • SAML metadata URL input and certificate upload
  • SSO enforcement toggle (require SSO for all org members)
  • Test connection button with status indicator
  • SSO user provisioning log

Backend Requirements

  • Supabase Auth SSO: Use built-in SAML support (supabase.auth.signInWithSSO)
  • API route: POST /api/sso/configure — save IdP metadata
  • API route: GET /api/sso/status — check SSO connection health
  • Supabase table: sso_configs (org_id, idp_type, metadata_url, cert, enforce, created_at)
  • Login page update: "Sign in with SSO" button for configured orgs

Supported Providers

  • Okta — SAML 2.0 (most common for enterprise)
  • Azure AD — SAML 2.0 + OIDC
  • Google Workspace — SAML 2.0
  • OneLogin — SAML 2.0
  • Generic SAML — any compliant IdP

Effort Estimate

  • UI View + Setup Wizard: 2-3 days
  • Supabase SSO config: 1-2 days
  • Login page SSO button: 0.5 day
  • Testing with Okta dev account: 1 day
  • Total: ~5-7 days

3. Custom ML Models / Intelligence View

Not Built Business Tier

UI Components to Build

  • MLModelsView — new view in page.jsx (~400 lines)
  • Model cards: Anomaly Detection, Revenue Prediction, Expense Categorization
  • Model performance dashboard (accuracy, precision, last trained)
  • Anomaly alerts feed with severity levels
  • Revenue prediction chart with confidence intervals
  • "Retrain Model" button (triggers Claude analysis in MVP)

Backend Strategy (Two Options)

  • Option A — Claude-Powered (Recommended for MVP):
  • Use Claude API to analyze financial data and generate "model outputs"
  • Anomaly detection = Claude analyzing transactions for outliers
  • Revenue prediction = Claude running trend analysis on historical data
  • Option B — Real ML (Future):
  • Python ML pipeline with scikit-learn or Prophet
  • Supabase Edge Function for model serving

Pre-Built Models to Show

  • Anomaly Detector: Flags unusual transactions, spend spikes, revenue drops
  • Revenue Forecaster: ML-enhanced 12-month projection (vs. formula-based in Forecast view)
  • Expense Classifier: Auto-categorizes transactions into GL accounts
  • Churn Predictor: Identifies at-risk revenue streams

Effort Estimate

  • UI View + Model Cards: 2-3 days
  • Claude API integration for "models": 2-3 days
  • Anomaly detection prompts + results UI: 1-2 days
  • Demo data for model outputs: 1 day
  • Total: ~7-10 days

4. Slack Integration (Complete Build-Out)

Shell Only Growth Tier

UI Updates Needed

  • Replace static Slack card in IntegrationsView with real OAuth button
  • Channel selector after OAuth — pick where alerts go
  • Notification preferences panel (what events trigger alerts)
  • Weekly digest toggle and schedule picker
  • Connection status indicator with last-synced timestamp

Backend Requirements

  • Slack App created in api.slack.com (bot + incoming webhooks)
  • OAuth 2.0 flow: /api/integrations/slack/authorize → /api/integrations/slack/callback
  • Supabase table: slack_connections (org_id, team_id, access_token, channel_id, webhook_url)
  • Webhook sender: POST to Slack on financial events
  • Cron job: Weekly financial digest message

Notification Types

  • Cash balance alerts (below threshold)
  • Revenue milestone notifications
  • Month-end close progress updates
  • Anomaly detection alerts (from ML view)
  • Weekly financial health digest

Effort Estimate

  • Slack App setup + OAuth flow: 1-2 days
  • Webhook sender service: 1 day
  • UI updates to IntegrationsView: 1 day
  • Weekly digest cron: 0.5 day
  • Total: ~4-5 days
Dashboard Impact: No new view needed — updates happen within existing IntegrationsView. Replace the static Slack card with a real OAuth-powered connection card. Add notification preferences to Settings view.
