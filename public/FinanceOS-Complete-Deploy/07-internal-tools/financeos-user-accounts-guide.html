User Accounts, Roles & Dashboard Assignment

How new team members get invited, what account type they receive, and what dashboard experience loads for each role and plan tier.
Castford Platform Architecture · March 2026
01 The Invitation Flow

New users join Castford through the team invitation system. Only Owners and Admins can invite new members. Each invite includes a specific role assignment that determines what that user can see and do.

Detailed Sequence: Invite → Accept → Dashboard Load
Admin
Clicks "+ Invite User" in Admin Console, enters email and selects role from dropdown: Owner | Admin | Manager | Budget Owner | Viewer
Frontend
Sends POST /api/team/invite with { email, invite_email, role }
API Route
Validates: caller must be Owner or Admin → checks seat limit (org.seats_used vs org.seats_limit) → checks if already invited → generates crypto.randomUUID() token valid 7 days
Supabase
Inserts into invitations table: org_id, invited_by, email, role, token, status: "pending", expires_at → Writes to audit_log with action: "INVITE"
Invitee
Receives email → clicks accept link → Supabase Auth creates account → user row created in users table with assigned role and org_id
Dashboard
PlanProvider calls GET /api/plan?email= → Stripe checkpoint resolves org's plan tier → canAccessView() determines which nav items are active vs locked → role determines write/edit permissions within views
API: POST /api/team/invite
// Request body { "email": "admin@company.com", // who is inviting (must be Owner or Admin) "invite_email": "new@hire.com", // who is being invited "role": "viewer" // role to assign (default: "viewer") } // Validations performed: // 1. Caller lookup: user must exist in users table // 2. Permission check: user.role must be "owner" or "admin" // 3. Seat limit: org.seats_used must be < org.seats_limit // 4. Duplicate check: no pending invite for same email+org // 5. Token generation: crypto.randomUUID(), expires in 7 days // Response { "success": true, "invitation": { "id", "token", "email", "role", "status": "pending", "expires_at" }, "message": "Invitation created for new@hire.com" }
🔒
Seat Limits by Plan Tier: Free = 1 seat · Starter = 5 seats · Growth = 25 seats · Business = Unlimited. When the limit is reached, the API returns a 403 with a message to upgrade. The admin sees this in the UI as a disabled invite button.
02 The 5 Account Types (Roles)

Every user has a role stored in the users.role column. The role determines what actions they can take within the features their plan unlocks.

👑 Owner

Owner

The account creator / org founder. Full platform control. Typically the CFO, CEO, or VP of Finance. Only one per org (though can be transferred).
  • ✓ Full read/write access to all views
  • ✓ Manage billing, upgrade/downgrade plans
  • ✓ Invite users, assign any role
  • ✓ Change other users' roles
  • ✓ Delete org, rotate API keys
  • ✓ Configure SSO/SAML, integrations
  • ✓ Access SOX audit trails
  • ✓ Export all data, generate compliance reports
Example: Alex Morgan — CEO & Founder
🛠 Admin

Admin

Trusted second-in-command. Can manage team and most settings. Cannot change billing or delete the org. Typically VP Finance or FP&A lead.
  • ✓ Full read/write access to all views
  • ✓ Invite users, assign roles (except Owner)
  • ✓ Configure integrations, Slack
  • ✓ Manage close tasks, approve workflows
  • ✓ Access audit trails
  • ✗ Cannot change billing or plan
  • ✗ Cannot delete organization
  • ✗ Cannot change Owner's role
Example: Sarah Chen — VP of Finance
📊 Manager

Manager

Department-level access. Can create and edit financial data, run scenarios, manage their team's close tasks. Cannot invite users or change settings.
  • ✓ Read/write P&L, forecasts, scenarios
  • ✓ Create and complete close tasks
  • ✓ Use AI Copilot
  • ✓ Export reports
  • ● View-only on team management
  • ✗ Cannot invite users
  • ✗ Cannot configure integrations
  • ✗ Cannot access SSO or audit settings
Example: Priya Patel — Controller
💰 Budget Owner

Budget Owner

Scoped to their department's budget and data. Can edit budgets they own, view P&L related to their dept, and manage their assigned close tasks.
  • ✓ Read/write their department's budgets
  • ✓ View P&L (filtered to their scope)
  • ✓ Complete their assigned close tasks
  • ✓ Use AI Copilot (scoped)
  • ● Read-only on forecasts and scenarios
  • ✗ Cannot modify org-wide data
  • ✗ Cannot access admin console
  • ✗ Cannot export full datasets
Example: James Rodriguez — Operations
👁 Viewer

Viewer

Read-only access. Sees dashboards and reports but cannot modify anything. Perfect for board members, investors, or cross-functional stakeholders.
  • ✓ View all dashboards (per plan tier)
  • ✓ View KPIs and reports
  • ✓ Use AI Copilot (read-only queries)
  • ✗ Cannot edit any data
  • ✗ Cannot create tasks or scenarios
  • ✗ Cannot configure anything
  • ✗ Cannot invite users
  • ✗ Cannot export data
Example: David Kim — Revenue Analyst

03 Plan Tiers & What Each Dashboard Unlocks

The plan tier (set at the org level via Stripe) determines which views/features appear in the sidebar. The role determines what a user can do within those views. These are two separate layers of access control.

⚡
Two-Layer Access Model: Plan Tier (org-wide, from Stripe) controls which sidebar items are accessible. Role (per-user) controls read/write/admin permissions within those items. A Viewer on the Business plan sees all 17 views but can only read. An Owner on Free sees only the Dashboard overview but has full write access.

05 How Dashboard Loading Works (Technical Flow)
On Every Dashboard Load:
1
PlanProvider mounts
Wraps entire dashboard. Calls GET /api/plan?email={user.email} with cache: "no-store". No caching — Stripe is checked every time.
2
Stripe Checkpoint resolves plan
API route looks up user's email in Stripe customer list → checks active subscriptions → maps Stripe product ID to plan key via PRODUCT_PLAN_MAP → returns plan tier + feature list.
3
PlanContext populated
React context stores: { plan, features[], loading, subscription }. All child components can call usePlan() to read the current tier.
4
Sidebar renders with lock badges
Each NAV_ITEM is checked via canAccessView(verifiedPlan, item.id). If false, a lock badge appears with the required plan label from requiredPlanForView(item.id).
5
View content or UpgradePrompt renders
When a user clicks a nav item: if canAccessView() returns true → full view component renders. If false → UpgradePromptInline shows with the plan name + price + CTA to upgrade.
6
Role-based write permissions applied
Within each unlocked view, the user's role from the users table determines whether edit/create/delete buttons appear. Viewers see read-only. Budget Owners see scoped edit. Owners/Admins see everything.

07 Database Schema (Supabase Tables)
users
id uuid PK email text UNIQUE full_name text role text (owner|admin|manager|budget_owner|viewer) org_id uuid FK → organizations avatar_url text last_sign_in_at timestamptz created_at timestamptz
invitations
id uuid PK org_id uuid FK → organizations invited_by uuid FK → users email text role text (role to assign on accept) token uuid (unique accept link) status text (pending|accepted|expired) expires_at timestamptz (7 days from create) created_at timestamptz
organizations
id uuid PK name text plan text (free|starter|growth|business) stripe_customer_id text seats_used integer seats_limit integer created_at timestamptz
audit_log
id uuid PK org_id uuid FK → organizations user_id uuid FK → users action text (INVITE, CREATE, UPDATE, etc.) resource_type text resource_id text metadata jsonb ip_address text created_at timestamptz
